State-of-the-art LLMs now operate with context windows of 200K to 1M tokens (roughly 300–1,500 pages). Claude Opus 4.6 and Sonnet 4.6 support 1M tokens at GA; Gemini 3.1 Pro matches this. Commands

MCPs as Context

Write an MCP server or a top-level document describing what your service can do — its inputs, outputs, and capabilities. This gives AI agents a way to understand and interact with your service without loading its internals into context. For common tools (GitHub, Jira, Slack), pre-built MCPs already exist — just connect them.

Compensate for missing context with precise prompts. Reference file paths, state constraints explicitly, and point to existing patterns as examples.

🚫

Ignore Files

Control what the AI doesn't see. Most AI tools support ignore files that work like .gitignore — exclude build artifacts, vendored deps, legacy modules, and generated code.

.cursorignore .claudeignore .aiderignore /dist /node_modules *.designer.cs
🗜️

Context Compaction

Cursor and Claude Code auto-compact at ~64–75% capacity, summarizing earlier context to free space. When this triggers, fidelity drops — earlier file reads, decisions, and errors lose influence. Plan for it: break work into focused sessions, don't let one session sprawl across unrelated tasks.

~147K effective ceiling 30–40K tokens reserved for system Auto-triggers silently
🔀

Subagents & Task Decomposition

The most powerful pattern for fighting context limits: spawn child agents with scoped context. Claude Code's Task system and Cursor's subagents each get fresh context windows. Parent dispatches, children execute in isolation, results bubble up.

Parent → Child pattern Scoped context per subtask Parallel execution
💰

Cost Awareness

More context = more tokens = more money and slower responses. A 900K request costs the same per-token but you're burning 4.5x the tokens. Even on flat-rate plans, wasted context means earlier compaction and degraded output quality. Lean context is fast context.

Token cost scales linearly Latency scales too Compaction = lost fidelity

Two Kinds of Context Documents

Not all documentation serves the same purpose. Martin Fowler's "Context Anchoring" distinguishes two layers — and most teams only do the first.

📘 Priming Document

Stable · Shared · Updates quarterly

"Here's how this project works." Tech stack, architecture patterns, naming conventions, code examples. This is what CLAUDE.md already is. Every developer, every session, same document.

CLAUDE.md Tech stack Architecture Conventions

📒 Feature Document

Volatile · Per-feature · AI-maintained

A running document where the AI succinctly describes what changes it made and why. After each session, the AI appends a summary of decisions, rejected approaches, and current state. We will add a Cursor command to auto-write to this file. This is what keeps context alive between sessions.

Changes made Decisions & rationale Rejected approaches Cursor command to write
Fowler's litmus test: If you feel anxious closing a chat session because you'll lose context, your decisions aren't properly anchored outside the conversation. The feature document solves this.

💡 Real-World Example — Local MCPs in a Monolith

Why "Cost" Should Be an MCP, Not Context

Right now, if an AI agent needs to do anything that touches cost in a large monolith, it has to load a bunch of cost-related files into context, understand the domain model, figure out the relationships, and reason about it from scratch every session. That's expensive context-wise and error-prone in a monolith that big.

Imagine instead that cost is exposed as a local MCP server — a thin facade over the existing codebase. The agent doesn't need to understand how cost works internally. It calls task-level tools and gets answers. The cost domain knowledge lives inside the MCP server implementation, not in the AI's context window.

Without MCP: Cost in Context

Agent loads 40+ cost files into context every session

Has to re-learn the domain model, relationships, and business rules from scratch

Burns context tokens on code it doesn't need to modify — just query

Gets it wrong when the module is too large to fit alongside the actual task

With MCP: Cost as a Service

Agent calls get_cost_breakdown(project_id)

Agent calls get_budget_vs_actual(project_id, period)

Agent calls get_change_order_impact(change_order_id)

Zero cost files in context. Domain knowledge lives in the MCP server.

The key insight: You don't have to decompose the monolith to get the AI benefits of service boundaries. Build a thin MCP facade over domains the AI consumes frequently — cost, reporting, project status, document management. The monolith stays as-is, but the AI's interface becomes clean task-level tools. Use MCP for AI-as-user (querying cost data). Keep source code in context for AI-as-developer (changing how cost works).

Key Takeaway

The teams that get the most out of AI tooling aren't the ones with the best prompts.
They're the ones with the best-organized codebases.
